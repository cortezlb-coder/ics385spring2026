Project Overview
Maui Surf House is a vacation bnb located in Kihei on Maui. The goal of the project is to give surfers coming from Australia a fast way to understand the property, review surf-friendly amenities, and book a stay. The amenities will include surfboard storage or rent-a-board service, outdoor showers for post-beach cleanup, and optional rental vehicles like Jeeps or Tacomas equipped to haul gear.

The application will stay simple in booking and will include multiple APIs to support the visitor experience. The page will include local food recommendations, a surf cam, weather reports, and night events near the property. The application will have 5 pages consisting of the marketing page showcasing the property and amenities, booking page, property management via login, customer login and interface, and splash page. The goal is to make the interface clear and quick for both the customer and admin. The focus is on usability, simplicity, and real-world application of a full-stack system using React, Express, and MongoDB.

For Week 14, the admin portion of the project will be protected with username and password authentication using Passport.js local strategy. Admin access will require a valid login session before property edits can be made, and the login screen will be designed as a secure gateway into the property management dashboard.

Data Flow
Data is stored in MongoDB and managed through an Express backend. The React front-end sends requests to the backend to retrieve or update property information. The Express server processes the request and interacts with the database. The returned data is then displayed in the React components.

Authentication flow for admin access:
- A user opens the admin login screen and enters a username or email plus password.
- Passport.js local strategy checks the submitted credentials against the stored admin record in MongoDB.
- Passwords are compared with bcrypt hashes instead of plain text passwords.
- If credentials are valid, Express creates a session and the admin can access protected routes.
- If credentials are invalid, the system returns a generic error message and does not reveal whether the username exists.
- The admin session remains active until logout or session expiration.

The customer-facing pages remain public, but any route that changes property data must be protected by authentication middleware.

Mongoose Schema Details
Property schema fields:
- name: String
- island: String
- type: String
- description: String
- amenities: String[]
- targetSegment: String
- imageURL: String
- reviews: Review[]

Review schema fields:
- guestName: String
- rating: Number, required, min 1, max 5
- comment: String
- date: Date

User schema fields for authentication:
- username: String, required, unique
- email: String, required, unique
- passwordHash: String, required
- role: String, default "admin"
- createdAt: Date

Authentication schema notes:
- Plain text passwords are never stored in MongoDB.
- Passwords are hashed before save using bcrypt.
- The user record is used only for login, session creation, and route protection.

How the schema maps to the UI:
- name, island, imageURL map to the Hero component.
- description and targetSegment map to the About component.
- amenities map to the Amenities component and the surf support details on the marketing page.
- reviews map to the property details and review counts shown on the backend-rendered property page.
- rating validation supports the review form and API test cases.
- username and password map to the admin login screen.
- role and session state map to the protected admin dashboard and logout flow.

Authentication and Security Requirements
Admin Login Behavior
- Admin users must sign in with username or email plus password before accessing protected management pages.
- Invalid credentials must return a generic error message.
- Successful login must redirect to the protected admin dashboard.
- Logout must destroy the session and return the user to the login page.

Password and Session Security
- Passwords must be stored as bcrypt hashes, not plain text.
- Session cookies must be signed with a secret stored in environment variables.
- Protected routes must check whether the request is authenticated before rendering or saving data.
- Login errors should not reveal whether a specific username or email exists.
- Session data should be stored in MongoDB so the login survives page refreshes.

Access Control Rules
- Marketing, splash, and booking pages remain public.
- Admin dashboard pages are private and require an active session.
- Any route that edits property content must reject unauthenticated requests.
- Customer booking access stays separate from admin property editing access.

Acceptance Criteria (UPDATED)
All Pages
- Each page loads in under 2 seconds with no errors.
- UI displays correctly with no missing or undefined data.
- API responses return expected status codes and do not crash the application.
- Application runs without crashes or broken UI.

Splash Page
- User can enter the main site with 1 click.

Marketing Page
- Displays property name, image, description, location, and target segment.
- Shows at least 3 surf-related amenities: board storage or rental, outdoor shower, and Jeep or Tacoma option.
- User can reach the booking CTA on the same page without navigation.
- User can understand the property and find booking in under 10 seconds.
- GET /properties returns status 200 and includes one seeded property named Maui Surf House with island equal to Maui.
- GET /properties?format=json returns JSON with name, island, type, description, amenities, targetSegment, imageURL, and reviews fields.

Booking Page
- User can select dates and submit booking.
- Form blocks submission if required fields are missing.
- Valid submission shows a confirmation message.
- Invalid date ranges or missing guest information show an error message instead of creating a booking.

Customer Dashboard
- Valid login allows access; invalid login shows error message.
- User can view booking information after login.
- Customer dashboard only displays the logged-in customer's booking data.

Admin Dashboard
- Admin can update property details including description, amenities, and image.
- Changes appear on the site after page reload.
- Invalid inputs are blocked or show error message.
- Admin login requires a valid username or email plus password before access is granted.
- Invalid admin login attempts return a generic error message and do not expose whether the account exists.
- Protected admin routes reject unauthenticated requests and redirect to the login page or return a forbidden response.
- A successful admin login creates a session that survives page refreshes until logout or expiration.
- Logging out destroys the session and removes access to the protected dashboard.

Visitor Statistics Dashboard
- Displays at least one chart, island selector, weather widget, and summary value.
- Island selector includes at least 4 islands and updates data in under 1 second.
- Chart updates based on selected island and shows clear labels.
- Weather updates based on selected island and shows a fallback message if unavailable.

Security Acceptance Criteria
- Passwords are hashed with bcrypt before being stored in MongoDB.
- Plain text passwords are never stored, displayed, logged, or returned in API responses.
- Session secrets and other sensitive values are kept in environment variables and not hard-coded in the source.
- Session cookies are signed and protected using secure Express session settings.
- Session data is stored in MongoDB so authenticated sessions survive page refreshes.
- All routes that create, update, or delete property data require authentication.
- Admin-only routes reject users who are not signed in as an admin.
- Customer routes reject unauthenticated users and prevent access to another customer's booking information.
- Login and form validation errors use safe messages that do not reveal private account details.
- Server-side validation is used for required fields, ratings, dates, image URLs, and editable property content.
- Invalid API requests return clear 400, 401, or 403 responses instead of exposing stack traces.
- User input is sanitized or safely rendered to reduce cross-site scripting risk.
- MongoDB queries use Mongoose models and validation instead of trusting raw client input.
- Logout destroys the server session and prevents access to protected pages using the browser back button.

Overall Experience
- User can understand the property and reach booking in under 10 seconds.
- Application runs without crashes or broken UI.
